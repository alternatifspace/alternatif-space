## 1. Overview

alternatif.space is a civic technology platform structured across four subdomains: partai (party formation), mufakat (discussion), simposium (live deliberation), and perpus (documentation). Party membership on partai acts as a persistent ideological flag carried across all subdomains.

This TRD covers the technical implementation across all four subdomains. partai is built first. The other subdomains follow once the previous phase is validated.

The TRD implements the PRD (v0.3) across Phase 0, Phase 1, and Phase 2. Only Phase 0 is built first. Phases 1 and 2 are documented here for architectural foresight — no Phase 1 or 2 tables, functions, or components are built until the previous phase is validated.

## 2. Tech Stack

| Layer | Choice | Notes |
|---|---|---|
| Framework | SvelteKit | All four subdomains. One framework, one mental model. |
| Styling | Tailwind CSS | Utility-first, mobile-first |
| Auth | Clerk | JWT scoped to `.alternatif.space` for cross-subdomain SSO |
| Database | Supabase (PostgreSQL) | RLS enforced at database level |
| Storage | Supabase Storage | Party logos, share cards, future assets |
| Rich Text Editor | TipTap (Svelte) | Manifesto creation and editing |
| Diff Library | diff-match-patch | Manifesto amendment diffs (Phase 1) |
| Web Deployment | Cloudflare Pages | Best free tier; SE Asia edge nodes. One deployment per subdomain. |
| Native Wrapper | Capacitor | Single app wrapping all four subdomains for app store distribution |
| Android Distribution | Play Store via Capacitor | One listing. Trust signal for Indonesian users. |
| iOS Distribution | App Store via Capacitor | One listing. Apple review required. |
| Push Notifications | Capacitor (FCM + APNs) | Web Push as fallback for browser users |
| Shared UI | `packages/ui` (Svelte components) | Shared across all four SvelteKit apps |
| Automation | GitHub Actions | Scheduled cron jobs for vote closing and dormant status |

**Why SvelteKit for all subdomains (previously Astro for mufakat and perpus):**
The original multi-framework decision predated the native app distribution requirement. Four separate Capacitor apps — one per subdomain — would require four separate app store listings and force users to download multiple apps, which kills adoption. A single Capacitor app requires a single framework. SvelteKit handles SSR, static output (`adapter-static` for perpus), and lazy hydration well enough that Astro's island architecture advantage does not justify the added complexity. One framework, one Capacitor app, one app store listing per platform.

A single `native/` Capacitor app wraps all four subdomains under one binary. Navigation between subdomains happens within the app — no external browser required. One Play Store listing, one App Store listing.

```
Each SvelteKit app builds independently to static files
  ↓
native/capacitor.config.ts points to the live web URLs (or bundled builds)
  ↓
npx cap sync copies assets into Android/iOS native projects
  ↓

```

The Capacitor shell uses in-app WebViews per subdomain with a native bottom tab bar or drawer navigation for switching between partai, mufakat, simposium, and perpus. This gives users a unified app experience while keeping subdomain codebases independent.

App Store submission frames the platform as a civic education app. Review timeline should be factored into launch planning — Apple review typically takes 1–3 days but can be longer for first submissions.

## 5. Auth & Cross-Subdomain Session

Clerk handles authentication. The JWT is scoped to `.alternatif.space` (root domain cookie) so a user authenticated on `partai.alternatif.space` is automatically authenticated on all other subdomains without re-login.

```
User logs in via Clerk
  ↓
Clerk issues JWT
  ↓
Cookie set: domain=.alternatif.space
  ↓
mufakat, simposium, perpus all read the same cookie
  ↓
Each subdomain initialises Supabase client with Clerk JWT
  ↓
Supabase RLS uses auth.uid() from the JWT
```

Clerk JWT template must be configured to produce a Supabase-compatible token with `sub` mapped to the user's Supabase UUID.

**Party membership flag in other subdomains:**
Any subdomain determines a user's current party with a single query:
```sql
SELECT party_id FROM party_members
WHERE user_id = auth.uid() AND status = 'active'
LIMIT 1;
```
No special sync or replication needed.

## 7. Row Level Security

All tables have RLS enabled. Core policies for Phase 0:

```sql
-- Parties: public read, authenticated create, leader-only update
ALTER TABLE parties ENABLE ROW LEVEL SECURITY;

CREATE POLICY "parties_public_read" ON parties
  FOR SELECT USING (true);

CREATE POLICY "parties_authenticated_insert" ON parties
  FOR INSERT WITH CHECK (auth.uid() IS NOT NULL);

CREATE POLICY "parties_leader_update" ON parties
  FOR UPDATE USING (leader_id = auth.uid());

-- Party members: public read, complex writes via Edge Functions
ALTER TABLE party_members ENABLE ROW LEVEL SECURITY;

CREATE POLICY "members_public_read" ON party_members
  FOR SELECT USING (true);

-- Party posts: members-only read (Phase 1)
ALTER TABLE party_posts ENABLE ROW LEVEL SECURITY;

CREATE POLICY "posts_members_only_read" ON party_posts
  FOR SELECT USING (
    EXISTS (
      SELECT 1 FROM party_members
      WHERE party_id = party_posts.party_id
        AND user_id = auth.uid()
        AND status = 'active'
    )
  );

CREATE POLICY "announcements_leader_only_insert" ON party_posts
  FOR INSERT WITH CHECK (
    type != 'announcement' OR
    EXISTS (
      SELECT 1 FROM parties
      WHERE id = party_id AND leader_id = auth.uid()
    )
  );
```

## 8. Dormant Status Logic

`status` is a stored value in the database — the source of truth for all subdomains. Two mechanisms keep it current:

**Trigger on `leader_last_active_at`** handles real-time cases — when a leader logs in, their activity timestamp updates and the trigger immediately re-evaluates dormant status:

```sql
CREATE OR REPLACE FUNCTION update_party_dormant_status()
RETURNS TRIGGER AS $$
BEGIN
  UPDATE parties
  SET status = CASE
    WHEN NEW.leader_last_active_at < now() - interval '30 days'
      AND status = 'active' THEN 'dormant'
    WHEN NEW.leader_last_active_at >= now() - interval '30 days'
      AND status = 'dormant' THEN 'active'
    ELSE status
  END
  WHERE id = NEW.id;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

**Daily GitHub Actions cron** handles the cold case — leaders who go silent and never trigger the above:

```yaml
# .github/workflows/maintenance.yml
on:
  schedule:
    - cron: '0 0 * * *'        # daily at midnight UTC
jobs:
  update-dormant-status:
    runs-on: ubuntu-latest
    steps:
      - run: |
          curl -X POST \
            ${{ secrets.SUPABASE_FUNCTIONS_URL }}/update-dormant-status \
            -H "Authorization: Bearer ${{ secrets.SUPABASE_SERVICE_KEY }}"
```

## 9. Edge Functions

Complex writes that touch multiple tables or require threshold checks go through Supabase Edge Functions, not direct client calls.

### Phase 0 Functions

**`join-party`**
Checks user has no current active membership, inserts into `party_members`, writes to `party_membership_history`.

**`leave-party`**
Removes from `party_members`, writes leave record to `party_membership_history`.

**`generate-invite-token`**
Creates a cryptographically random token, inserts into `party_invites` with 7-day expiry.

**`use-invite-token`**
Validates token (exists, not used, not expired), triggers join flow, marks token used.

**`submit-application`**
Inserts application. Checks user has no existing pending application for the same party.

**`review-application`**
Leader approves or rejects. On approval triggers join flow. Checks caller is party leader or delegated role.

**`update-dormant-status`**
Called by daily GitHub Actions cron. Updates all parties where `leader_last_active_at` has crossed the 30-day threshold.

### Phase 1 Functions

**`cast-vote`**
Inserts vote, checks if threshold is now met, if so closes proposal and executes outcome atomically (manifesto update, leadership transfer, etc.).

**`sign-petition`**
Inserts signature, checks if threshold is met, if so auto-creates a recall vote proposal.

**`close-expired-proposals`**
Called every 15 minutes by GitHub Actions cron. Tallies results on any proposal past `voting_ends_at`, updates status, executes outcome.

**`send-push-notification`**
Called internally when a notification is created. Looks up user's `push_subscriptions`, dispatches via FCM (Android), APNs (iOS), or Web Push (browser fallback).

### GitHub Actions Cron — Single Workflow File

Both scheduled jobs live in one workflow:

```yaml
# .github/workflows/maintenance.yml
on:
  schedule:
    - cron: '*/15 * * * *'     # every 15 minutes — vote closing
    - cron: '0 0 * * *'        # daily — dormant status update
jobs:
  close-expired-proposals:
    runs-on: ubuntu-latest
    steps:
      - run: |
          curl -X POST \
            ${{ secrets.SUPABASE_FUNCTIONS_URL }}/close-expired-proposals \
            -H "Authorization: Bearer ${{ secrets.SUPABASE_SERVICE_KEY }}"

  update-dormant-status:
    runs-on: ubuntu-latest
    steps:
      - run: |
          curl -X POST \
            ${{ secrets.SUPABASE_FUNCTIONS_URL }}/update-dormant-status \
            -H "Authorization: Bearer ${{ secrets.SUPABASE_SERVICE_KEY }}"
```

### State Management

Two global Svelte stores cover the majority of shared state. These live in `packages/ui` so all four subdomains can import them:

```javascript
// packages/ui/stores/currentUser.js
export const currentUser = writable(null);
// { id, display_name, bio, last_active_at }

// packages/ui/stores/currentParty.js
export const currentParty = writable(null);
// { id, slug, name, logo_url, status, role: 'leader' | 'member' }
```

Both populated on app initialisation after Clerk session resolves. Components read from stores — they do not fetch independently.

### Create Party Flow (Multi-step) *(updated v1.2)*

Step 1 (starting point) → Step 2 (parameters) → Step 3 (identity) → Step 4 (council) → Step 5 (review) → Publish

State held in a single local store for the duration of the flow. Nothing written to the database until the final publish step. On publish, `governance_config` is frozen and `honeymoon_ends_at` is set to `now() + interval '3 months'`.

**Step 1 — Four options:**
- Vanguard, Republic, Commune: pre-fill the local store's parameter fields with preset values, then navigate to Step 2
- Custom: navigate to Step 2 with all parameter fields in the local store left at `null`

**Step 2 — Two states depending on path:**
- Preset path: all fields populated, CTA active immediately, founder can proceed without touching anything
- Custom path: all fields `null`, CTA disabled, enabled only once every parameter field has a non-null value. A derived store counts unset fields and surfaces the completion counter in the UI.

```javascript
// Derived store for Custom path completion
const unsetCount = derived(createPartyStore, ($s) =>
  Object.values($s.governance_config).filter(v => v === null).length
);
// CTA disabled when unsetCount > 0
```

**Archetype is never written to the store or database.** It is used only to determine which pre-fill values to apply in Step 1.

### Manifesto Diff (Phase 1)

Amendment proposals render a diff using `diff-match-patch` against plain text extracted from TipTap JSON. A "lihat versi lengkap" toggle switches to side-by-side view for members who find the diff hard to read.

```javascript
import { diff_match_patch } from 'diff-match-patch';

const dmp = new diff_match_patch();
const diffs = dmp.diff_main(
  tipTapToPlainText(currentManifesto),
  tipTapToPlainText(proposedManifesto)
);
dmp.diff_cleanupSemantic(diffs);
// render with red/green highlighting
```

## 11. Push Notifications

**Primary:** Capacitor Push Notifications plugin using FCM (Android) and APNs (iOS). Delivers to users whether or not the app is open.

**Fallback:** Web Push API via Service Worker for users accessing via browser who have not installed the app. Uses the same `push_subscriptions` table with `platform = 'web'`.

**Phase 1 launch:** In-app notification bell only (reads from `notifications` table). Push delivery added as fast follow after Phase 1 ships — `push_subscriptions` table is in the schema from day one to avoid a migration.

**Notification weights (in-app bell):**

| Type | Weight |
|---|---|
| Announcement | High |
| Recall vote opened | High |
| Petition threshold reached | High |
| Pulse check (with deadline) | High |
| Direct message | High |
| Position brief | Medium |
| Discussion reply | Low — active participants only |

## 14. Content Moderation

**Phase 0:** Manual approval for all new party creations. Moderator reviews party name, tagline, and manifesto before the party appears in browse. Target review time: 24 hours.

**Compliance baseline:** ITE Law — no blasphemy, no ethnic hate speech, no violence incitement. These are the only hard-coded rules. All other content rules emerge through the platform's own constitutional process.

Moderation queue is a Supabase dashboard view filtered on `parties.status = 'pending_review'`. No custom admin UI for Phase 0.

| Q2 | Dormant status — read-time vs stored | Stored value; Postgres trigger on activity + daily GitHub Actions cron |
| Q3 | Manifesto diff format | `diff-match-patch` on plain text; side-by-side toggle as fallback |
| Q4 | Push notifications | Capacitor (FCM/APNs) primary; Web Push fallback for browser users; in-app bell for Phase 1 launch with push as fast follow |
| Q5 | PWA vs native | PWA + Capacitor wrapper for Play Store and App Store distribution |
| Q6 | Multi-framework vs SvelteKit-only | SvelteKit for all four subdomains. Four separate Capacitor apps = four app store listings = unacceptable user friction. One Capacitor shell requires one framework. Astro's island architecture advantage does not outweigh the cost. |
| Q7 | Archetype label visibility | Archetype label (Vanguard / Republic / Commune / Custom) is never stored, never returned by any API, and never shown publicly. It is a UI-only founding tool that pre-fills the parameter form. Only the resulting raw `governance_config` values are stored and displayed. Filtering by archetype on the browse page is therefore impossible and not implemented — browse filters use membership model and activity status instead. |
